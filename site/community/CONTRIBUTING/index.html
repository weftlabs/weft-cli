    Contributing to Weft¶

    Thank you for your interest in contributing!

    Development Setup¶

    Prerequisites¶

    • Python 3.11 or higher
    • Git
    • Anthropic API key

    Setup Steps¶

    1. Clone repository: 

      git clone https://github.com/weftlabs/weft-cli.git
cd weft-cli

    2. Create virtual environment: 

      python -m venv venv
source venv/bin/activate  # On Windows: venv\Scripts\activate

    3. Install in development mode: 

      pip install -e ".[dev]"

    4. Configure environment: 

      cp .env.example .env
# Edit .env and add your CLAUDE_API_KEY

    5. Run tests: 

      pytest

    Code Style¶

    Python Style Guide¶

    • Follow PEP 8
    • Use Black for formatting
    • Use type hints for all functions
    • Write docstrings for all public APIs

    Format code: 

    black src/ tests/

    Check types: 

    mypy src/

    Docstring Format¶

    Weft uses minimal docstrings per the coding standards:

    """Configuration management for Weft projects."""

def load_weftrc(path: Path | None = None) -> WeftRC | None:
    """Load and validate .weftrc.yaml file."""
    # Implementation...

    Rules: - Module-level: 1 sentence describing purpose - Public functions: 1-2 lines max, only if intent not obvious - NO Args/Returns/Raises sections - NO examples in docstrings (use tests instead) - Prefer clear naming over verbose documentation

    Testing Requirements¶

    Writing Tests¶

    • Unit tests: Test individual functions/classes
    • Integration tests: Test full workflows
    • Coverage: Aim for 90%+ coverage
    • No network calls: Mock external APIs

    Running Tests¶

    # Run all tests
pytest

# Run with coverage
pytest --cov=src/weft --cov-report=term-missing

# Run specific test
pytest tests/unit/test_config.py -v

# Run integration tests only
pytest tests/integration/ -v

    Test Structure¶

    def test_feature_name():
    """Test description."""
    # Arrange
    input_data = ...

    # Act
    result = function_under_test(input_data)

    # Assert
    assert result == expected_value

    Pull Request Process¶

    Before Submitting¶

    1. Create feature branch: 

      git checkout -b feature/your-feature-name

    2. Make changes:

    3. Write code
    4. Add tests

    5. Update documentation

    6. Run checks: 

      # Format code
black src/ tests/

# Lint code
ruff check src/ tests/

# Run tests
pytest --cov=weft

# Check types (optional - known issues)
mypy src/

    7. Commit: 

      git add .
git commit -m "feat: Add feature description"

    Commit Message Format¶

    Follow Conventional Commits:

    <type>: <description>

[optional body]

[optional footer]

    Types: - feat: New feature - fix: Bug fix - docs: Documentation only - test: Adding tests - refactor: Code refactoring - style: Code style changes - chore: Maintenance tasks

    Examples: 

    feat: Add database agent implementation
fix: Handle empty prompt edge case
docs: Update CLI reference with new command
test: Add integration test for feature workflow

    Creating Pull Request¶

    1. Push branch: 

      git push origin feature/your-feature-name

    2. Open PR on GitHub

    3. PR Checklist:

    4. Tests pass
    5. Coverage >= 90%
    6. Documentation updated
    7. CHANGELOG.md updated
    8. Type hints added

    9. Docstrings written

    10. Wait for review

    Common failures: - Black formatting: Run black src/ tests/ locally - Ruff linting: Run ruff check src/ tests/ --fix to auto-fix - Test failures: Run pytest locally to reproduce - Coverage too low: Add tests for uncovered code

    After Review¶

    • Address review comments
    • Push changes
    • Request re-review

    Issue Reporting¶

    Bug Reports¶

    Include: - Description of bug - Steps to reproduce - Expected behavior - Actual behavior - Environment (OS, Python version) - Error messages/logs

    Feature Requests¶

    Include: - Description of feature - Use case - Proposed API/interface - Examples

    Development Workflow¶

    Adding a New Agent¶

    See docs/AGENTS.md for agent specifications and docs/ARCHITECTURE.md for system design.

    Adding a New CLI Command¶

    1. Create command file: src/weft/cli/mycommand.py
    2. Implement using Click
    3. Register in main.py
    4. Add tests
    5. Update documentation

    Modifying Prompt Specs¶

    Prompt specs follow semantic versioning. See prompt-specs/ directory for current versions.
