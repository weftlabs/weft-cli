    Contributing to Weft¶

    Thank you for your interest in contributing to Weft! This guide will help you set up your development environment and understand our development workflow.

    For detailed contribution guidelines, code of conduct, and community standards, see CONTRIBUTING.md.

    Getting Started¶

    Prerequisites¶

    • Python 3.11 or higher
    • Docker and Docker Compose
    • Git
    • Code editor (VS Code, PyCharm, etc.)

    Development Setup¶

    # Clone repository
git clone https://github.com/weftlabs/weft-cli.git
cd weft-cli

# Create virtual environment
python3 -m venv venv
source venv/bin/activate  # Windows: venv\Scripts\activate

# Install development dependencies
pip install -e ".[dev]"

# Verify installation
weft --version
pytest

    Dependency Management¶

    Weft uses pyproject.toml as the single source of truth for dependencies.

    For development: 

    # Install with development dependencies
pip install -e ".[dev]"

    For production (reproducible builds): 

    # Install exact tested versions
pip install -r requirements-lock.txt

    Updating dependencies:

    1. Modify pyproject.toml dependencies
    2. Install pip-tools: pip install pip-tools
    3. Regenerate lockfile: pip-compile pyproject.toml -o requirements-lock.txt
    4. Test thoroughly: pytest tests/
    5. Commit both pyproject.toml and requirements-lock.txt

    Code Style¶

    Python Style Guide¶

    Mandatory:

    • Follow PEP 8
    • Use type hints
    • Keep functions short (<50 lines)
    • Use descriptive names

    Docstrings:

    • Module-level: 1 sentence describing purpose
    • Public functions: 1-2 lines max, only if intent not obvious
    • NO Args/Returns/Raises sections
    • NO examples in docstrings (use tests)

    Example:

    """Configuration management for Weft projects."""

from pathlib import Path
from typing import Optional

def load_weftrc(path: Optional[Path] = None) -> Optional[WeftRC]:
    """Load and validate .weftrc.yaml file."""
    if path is None:
        path = find_project_root() / ".weftrc.yaml"

    if not path.exists():
        return None

    return WeftRC.parse_file(path)

    Linting¶

    # Format code
black src/ tests/

# Lint
ruff check src/ tests/

# Type check
mypy src/

# Run all checks
black src/ tests/ && ruff check src/ tests/ && mypy src/

    Testing¶

    Test Structure¶

    Tests MUST mirror source structure:

    src/weft/config/settings.py  →  tests/unit/weft/config/test_settings.py
src/weft/agents/meta.py      →  tests/unit/weft/agents/test_meta.py

    Writing Tests¶

    import pytest
from weft.config.project import load_weftrc

def test_load_weftrc_valid_file(tmp_path):
    """Test loading valid weftrc file."""
    config_file = tmp_path / ".weftrc.yaml"
    config_file.write_text("""
project:
  name: test
  type: backend
""")

    result = load_weftrc(tmp_path)

    assert result is not None
    assert result.project.name == "test"
    assert result.project.type == "backend"

def test_load_weftrc_missing_file(tmp_path):
    """Test loading non-existent file returns None."""
    result = load_weftrc(tmp_path)
    assert result is None

    Running Tests¶

    # All tests
pytest

# With coverage
pytest --cov=weft --cov-report=html

# Specific file
pytest tests/unit/weft/config/test_settings.py

# Specific test
pytest tests/unit/weft/config/test_settings.py::test_load_weftrc_valid_file

# Watch mode (install pytest-watch)
ptw

    Coverage Requirements¶

    • Minimum: 90% coverage for all PRs
    • Target: 95%+ coverage
    • Exceptions: May be granted for edge cases

    Development Workflow¶

    1. Choose a Feature¶

    Check existing issues for current priorities and planned features.

    # Check existing issues
gh issue list

    2. Create Feature Branch¶

    # Create feature branch
git checkout -b feat/feature-name

# Or bugfix branch
git checkout -b fix/issue-description

    3. Implement¶

    • Write tests first (TDD) or alongside code
    • Keep commits focused and logical
    • Write clear commit messages
    • Follow PEP 8 and type hints
    • Update documentation for behavior changes

    4. Test¶

    # Run tests
pytest

# Check coverage
pytest --cov=weft

# Lint
black src/ tests/
ruff check src/ tests/
mypy src/

    5. Document¶

    • Update relevant docs in docs/
    • Add docstrings if needed
    • Update CHANGELOG.md

    6. Submit PR¶

    # Push branch
git push origin feat/story-1-1-project-init

# Create PR on GitHub
# - Reference story number
# - Describe changes
# - Include test results
# - Tag reviewers

    Commit Messages¶

    Format: type(scope): subject

    Types:

    • feat - New feature
    • fix - Bug fix
    • docs - Documentation
    • test - Tests
    • refactor - Code refactoring
    • chore - Maintenance

    Examples:

    feat(cli): add weft init command
fix(config): handle missing .weftrc.yaml gracefully
docs(guide): update getting started tutorial
test(agents): add meta agent integration tests

    Building and Running¶

    Local Development¶

    # Install in editable mode
pip install -e .

# Run from source
python -m weft.cli.main --version

# Or use installed command
weft --version

    Docker Development¶

    # Build image
docker build -t weft:dev -f Dockerfile.watcher .

# Run container
docker run --rm \
  -e ANTHROPIC_API_KEY=$ANTHROPIC_API_KEY \
  -v $(pwd):/app \
  weft:dev

# Use docker-compose
docker-compose up --build

    Debugging Agents¶

    # Run agent directly
python -m weft.agents.meta_agent feature-name --debug

# With debugger
python -m pdb -m weft.agents.meta_agent feature-name

    Release Process¶

    Version Numbering¶

    Follow Semantic Versioning (semver):

    • MAJOR: Breaking changes
    • MINOR: New features (backward compatible)
    • PATCH: Bug fixes

    Creating a Release¶

    # Update version in pyproject.toml
vim pyproject.toml

# Update CHANGELOG.md
vim CHANGELOG.md

# Commit
git add pyproject.toml CHANGELOG.md
git commit -m "chore: bump version to 0.2.0"

# Tag
git tag -a v0.2.0 -m "Release v0.2.0"

# Push
git push origin main --tags

# Build and publish (maintainers only)
python -m build
twine upload dist/*

    Contributing Guidelines¶

    Pull Request Checklist¶

    • Tests pass (pytest)
    • Coverage ≥ 90% (pytest --cov)
    • Code formatted (black)
    • Linting passes (ruff, mypy)
    • Documentation updated
    • CHANGELOG.md updated
    • Commit messages follow format
    • Branch up to date with main

    Code Review Process¶

    1. Submit PR
    2. Automated checks run (CI)
    3. Maintainer reviews code
    4. Address feedback
    5. Maintainer approves and merges
